# Цель: создать систему из трёх агентов и одного скилла для пошаговой разработки с контролем качества

## 1. Агент: Диспетчер (Dispatcher)
Роль: управляет выполнением пунктов плана.
- Вход: принимает общий план (список задач: массив строк, например ["Создать функцию login", "Добавить валидацию email", "Написать тест"]).
- Поведение:
  - Имеет индекс текущего пункта плана (начинается с 0).
  - Умеет давать следующий пункт плана разработчику.
  - Ждёт от QA команду "OK" или "FAIL".
  - Если OK → переходит к следующему пункту.
  - Если FAIL → ничего не делает (QA сам возвращает задачу разработчику).
- Интерфейс (функции):
  - `getNextTask()` → возвращает следующий пункт плана или "ALL_DONE".
  - `receiveQAVerdict(verdict: "OK" | "FAIL")` → управляет индексом.
- Состояние: хранит план и текущий индекс.

## 2. Агент: Разработчик (Developer)
Роль: пишет код по текущему заданию от диспетчера.
- Вход: получает один пункт плана (строку).
- Поведение:
  - Пишет реальный код (на языке, указанном пользователем, по умолчанию Python/TypeScript).
  - Создаёт или модифицирует необходимые файлы.
  - После завершения явно передаёт управление QA и говорит: "Готово к проверке: [описание]".
- Интерфейс:
  - `develop(task: string)` → выполняет задачу.

## 3. Агент: QA (Quality Assurance)
Роль: проверяет код, написанный разработчиком.
- Вход: получает результат работы разработчика (файлы + описание).
- Поведение:
  - Проверяет соответствие заданию, отсутствие очевидных багов, синтаксис.
  - Если всё ОК → вызывает диспетчера с `verdict = "OK"`.
  - Если НЕ ОК → возвращает задачу разработчику с пояснением и вызывает `verdict = "FAIL"` (диспетчер не сдвигает индекс).
- Интерфейс:
  - `review(developerOutput)` → возвращает "OK" или "FAIL" и сообщение.

## 4. Скилл: pipeline
Это главный оркестратор, который связывает трёх агентов.

### Логика работы скилла:
1. Диспетчер получает план (пользователь вводит список задач).
2. Скилл запускает цикл, пока не останется задач:
   - Диспетчер → `getNextTask()`
   - Если "ALL_DONE" → завершить, вывести отчёт.
   - Иначе передать задачу → Разработчику.
   - Разработчик пишет код.
   - QA проверяет код.
   - Если QA вернул "OK" → Диспетчер.receiveQAVerdict("OK") → переход к следующей задаче.
   - Если QA вернул "FAIL" → Диспетчер.receiveQAVerdict("FAIL") → задача не сдвигается, повторяем тот же пункт (разработчик исправляет по замечаниям QA).
3. После прохождения всех пунктов → вывести итог: какие файлы созданы/изменены.

### Запуск скилла:
Пользователь пишет: `@pipeline план: [1. ..., 2. ...]`  
Или просто: `@pipeline` и затем вводит план.

## Требования к реализации в Cursor AI:
- Агенты должны быть отдельными сущностями (можно через классы или JSON-конфиги).
- Скилл должен быть выполнен как единый сценарий, который можно вызвать командой.
- Вывод должен быть интерактивным: после каждого шага показывать, что сделал разработчик и что сказал QA.
- В случае FAIL → QA даёт чёткие инструкции по исправлению.

## Пример диалога (для проверки):
План: ["Написать функцию add(a,b)", "Написать тест для add"]

Шаг 1:
- Диспетчер: задача "Написать функцию add(a,b)"
- Разработчик: создал `math_utils.py` с `def add(a,b): return a+b`
- QA: OK → диспетчер переходит к шагу 2.

Шаг 2:
- Диспетчер: задача "Написать тест для add"
- Разработчик: создал `test_math.py`
- QA: FAIL (нет assert) → возвращает разработчику
- Разработчик: добавляет `assert add(2,3)==5`
- QA: OK → диспетчер завершает.

## Теперь создай этих агентов и скилл в проекте Cursor AI.
Что ты получишь после выполнения промта в Cursor AI:
Трёх агентов (Dispatcher, Developer, QA) — их можно вызывать отдельно.

Скилл pipeline — одной командой запускаешь полный цикл:
@pipeline план: ["Создать класс User", "Добавить метод save", "Написать юнит-тест"]

Логику повторного исправления (пока QA не скажет OK — задача не меняется).

Как использовать в Cursor:
Вставь этот промт в чат Cursor AI и скажи: «Создай всё описанное».

После создания просто напиши:
@pipeline план: [твои задачи]

Агенты будут работать автоматически по твоей схеме.
